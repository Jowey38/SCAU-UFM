## Key Learnings

- **Project:** SCAU-UFM
- **Description:** SCAU urban flood model. C++20 numerical core (Anisotropic DPM HLLC) with embedded SWMM and D-Flow FM 1D engines, fully coupled by `CouplingLib`.
- SWMM 5.2 API: `swmm_setValue(swmm_NODE_LATFLOW)` writes the API inflow buffer; `swmm_getValue(swmm_NODE_LATFLOW)` reads the *routed* value, so read-back is only meaningful after a routing step. `swmm_setValue(swmm_NODE_HEAD)` sets outfall stage (enables river->pipe backwater coupling). End-of-simulation is signalled by `swmm_step` resetting elapsed time to 0.
- SWMM solver keeps process-wide global state: at most one open project per process (`SwmmEngine` uses an atomic singleton guard).
- D-Flow FM BMI 1.0: `get_var` returns an engine-owned pointer valid only until the next BMI call; single-point writes use `set_var_slice`; resolve symbols with `decltype` against the vendored `bmi.h` so the ABI cannot drift silently.
- CouplingLib core mutation model: `apply_*` only enqueue `CouplingEvent`s; cell volume/h change only on `replay_pending()`. Mirror this for any new exchange path.
- MockDFlowFMEngine whitelists variables: `water_level`, `discharge`, `lateral_discharge`, `gate_opening` (project-standard names; the real BMI kernel may use `s1` etc. — keep variable names configurable at the driver/config level).
